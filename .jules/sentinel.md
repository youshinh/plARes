## 2026-03-02 - Prevent Path Traversal in File Operations\n**Vulnerability:** User IDs and Room IDs extracted from websocket URLs were passed unvalidated to `Path(...)` constructions, allowing arbitrary file system access.\n**Learning:** Implicit trust of WebSocket URL queries (`user_id`, `room_id`) led to dangerous string interpolation in filesystem paths.\n**Prevention:** Sanitize all identifiers used in filesystem paths utilizing a robust whitelist regex `re.sub(r'[^a-zA-Z0-9_-]', '_', val)` immediately prior to path construction.
## 2025-02-23 - NoSQL Path Traversal in Firestore
**Vulnerability:** Unsanitized `user_id` inputs were directly interpolated into Firestore document paths (`db.collection("users").document(user_id)`). This allowed malicious users to inject NoSQL path traversal characters (like slashes) to read or modify arbitrary unauthorized documents.
**Learning:** Because user-supplied values are used directly as primary keys in NoSQL paths without validation, they represent a high-severity path traversal vulnerability, especially when document creation functions lack parameter limits or schema checks.
**Prevention:** Always validate and sanitize external inputs (e.g., using `re.sub(r'[^a-zA-Z0-9_\-]', '', user_id)`) before interpolating them into database path references, guaranteeing that only expected characters (like alphanumeric values and hyphens/underscores) can construct a path.
