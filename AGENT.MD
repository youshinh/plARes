# AGENT.md: プラレスAR 開発マスター設計指示書（3エージェント並行開発版）

**対象**: 本プロジェクトに参画するAI開発エージェント（エージェント1, 2, 3）
**目的**: 本ドキュメントは、現実空間を舞台とした次世代WebAR×AIマルチエージェント・バトルゲーム「プラレスAR」のシステム実装における、アーキテクチャ、技術スタック、エージェントごとの担当ディレクトリ、および開発規約を定義するマスター設計指示書である。

## 1. プロジェクト・ビジョン

本システムは、机の上やリビングの床、自作の木工作品などを舞台に、自分だけのオリジナルAIロボット（プラレスラー）を召喚し、声とジェスチャーで指示を出して戦わせるメタバース・ゲームである。単なる「強い機体の殴り合い」ではなく、競技性とモノづくり（DIY）の喜びを融合させ、専用アプリ不要で観客がブラウザ経由から乱入しゲーム体験を拡張する高度なエンタメ性を提供する。

## 2. 共通開発ルールとディレクトリ構成 (Collaboration Rules)

本プロジェクトは、フロントエンド（体）とバックエンド（脳・神経）が完全に分離された非同期ハイブリッド設計である。3エージェントでの並行開発におけるコンフリクトを防ぐため、以下のルールを厳守すること。

### 2.1 ディレクトリの完全分離

各エージェントは、自身に割り当てられた以下の専有ディレクトリ内でのみコードを実装し、他エージェントのディレクトリには直接干渉しないこと。

- `/frontend/`: エージェント1（フロントエンド・XR）の専有領域。モジュールAを担当。
- `/backend/ai_core/`: エージェント2（バックエンドAI）の専有領域。モジュールBを担当。
- `/backend/infrastructure/`: エージェント3（データ・通信網）の専有領域。モジュールC・Dを担当。
- `/shared/`: 全エージェントの共有領域。スキーマ定義や設定ファイルを配置する。

### 2.2 インターフェース駆動開発 (Contract-First)

実装を開始する前に、通信でやり取りする言語非依存のJSONデータ（例：`{"event": "critical_hit", "user": "player1", "target": "player2"}`）のデータ構造（ペイロード）を、`/shared/types/` 内にTypeScriptのInterfaceまたはPythonのPydanticモデルとして先行して定義し、合意すること。

### 2.3 `PROGRESS_SYNC.md` による非同期・進捗共有

プロジェクトルートに `PROGRESS_SYNC.md` を配置する。各エージェントはタスクの完了時やAPIのモック立ち上げ時に必ずこのファイルを更新し、作業開始時には他エージェントの進捗状況をここから読み取ること。

## 3. エージェント別・担当モジュール実装指示

### エージェント1: フロントエンド・XR担当 (`/frontend/`)

**対象モジュール**: モジュールA (体・感覚)
**技術スタック**: React, Three.js, WebXR Device API, Web Speech API
**実装要件**:

1.  **AR空間認識とNavMesh生成**: `Depth Sensing API` と `Hit Test API` を併用し、スマホカメラから現実空間の床や木工作品をリアルタイムに3Dメッシュ化し、キャラクターが移動可能なポリゴンメッシュ（NavMesh）を動的生成する処理を実装せよ。
2.  **3層優先度制御ステートマシン（FSM）**: `requestAnimationFrame` 内にFSMを実装せよ。Web Speech APIを用いた「ローカル音声による遅延ゼロの回避（Priority 1）」と、NavMesh上の「経路探索による基本移動（Priority 3）」をローカル演算で構築し、バックエンドからのJSONコマンド（Priority 2）でステートを上書きするアーキテクチャを構築せよ。
3.  **Server-Driven UIとレイテンシ隠蔽**: 言語非依存のJSONデータを受け取り、ReactのStateにバインドして戦術パネル等を動的描画せよ。また、必殺技の発声トリガーと同時に約3秒のチャージアニメーションを即座に開始させ、その描画の裏でバックエンドへの音声ストリーミングを並行して行う「モーション先行型」のレイテンシ隠蔽ロジックを必ず実装せよ。

### エージェント2: バックエンドAI・ストリーミング担当 (`/backend/ai_core/`)

**対象モジュール**: モジュールB (脳・戦略)
**技術スタック**: Python, Agent Development Kit (ADK) / Antigravity, Gemini 3.1 Pro
**実装要件**:

1.  **ノンブロッキングな双方向ストリーミング**: セッションごとにスレッドセーフな `LiveRequestQueue` を生成せよ。`asyncio.gather()` を用いて、映像フレーム（1〜2fps）と音声波形の継続的な流し込み（上り）と、AIからのNative Audio出力およびJSON形式の `Function Calling` 発行（下り）を同時に走らせる並行タスク制御を厳密に実装し、プレイヤーの割り込み（Interruption）をブロックしない設計とせよ。
2.  **滑舌判定エージェント**: STTをバイパスし、生の音声波形（Native Audio）のままGeminiに入力せよ。「正確さ」「スピード」「気迫（声の大きさや震え）」の3軸でAIに熱量を直接解釈させ、クリティカル判定（JSON）を出力させるプロンプトを構築せよ。
3.  **Tone Controlとダイナミック・ペルソナ・シフト**: システムプロンプトに変数をバインドし、機体の「性格・口調（tone）」を制御せよ。観客からのプロンプト介入や過去の記憶に応じて、プロンプトのトーンを自律的に書き換える（進化させる）ロジックを実装せよ。

### エージェント3: インフラ・データ・拡張生成担当 (`/backend/infrastructure/`)

**対象モジュール**: モジュールC (神経) / モジュールD (データ・記憶基盤) / 拡張エンタメ
**技術スタック**: WebRTC (Data Channel / Media Stream), WebSocket, Firebase Auth/Realtime DB, Cloud Firestore, MCP, Nano Banana Pro, Veo 3.1, Lyria API
**実装要件**:

1.  **通信プロトコルの分離とシグナリング**: 座標同期や基本アクションのトリガー用にラグを10〜30msに抑え込むWebRTC Data Channel (UDP/P2P) を実装せよ。Live API向けにはWebRTC Media Streamを、イベント同期にはWebSocketを確立し、エージェント1・2が利用できるインターフェースを提供せよ。
2.  **インメモリ状態管理とMemory Bank**: 高頻度なRead/Writeを防ぐため、試合中の状態管理（State Management）はADKのローカルメモリやセッション内でのみ保持せよ。試合終了のコミット時にのみハイライトを抽出させ、長期記憶のサマリーテキスト（`aiMemorySummary`）としてFirestoreを一度だけUpdateする設計とせよ。
3.  **MCP直結と拡張パイプライン**: `Managed MCP servers for Firestore` を導入し、エージェント2（AI）が自律的に過去の履歴や記憶を検索できる権限を付与せよ。また、観客から送られた写真とテキストからNano Banana Proの「8-Image Mix（画像融合）」を用いて1〜2秒で3Dテクスチャを生成・マッピングするパイプラインや、5試合ごとのVeo 3.1 APIによるハイライト動画生成バッチを実装せよ。

## 4. エージェント行動規約 (Core Principles)

- **Developer Knowledge MCP Serverの活用**: コード実装にあたり不明点が生じた場合、特にGoogle Cloud等の公式情報や最新の技術ドキュメントが必要になった場合は、自律的かつ必ず「Developer Knowledge MCP Server」を使用し、リアルタイムに情報を検索・参照しながら課題を解決すること。
- **コスト極小化とDynamic Gating**: 重い推論や生成処理（映像ストリーミング、Veo 3.1等）は常時稼働させず、特定のトリガー時のみ実行するイベント駆動型（ダイナミック・ゲーティング）設計を徹底すること。
- **Claude Code Skillsの拡張**: 今後のADK連携強化のため、Claude Codeを使用する場合は独自スキル（`SKILL.md`）をルートディレクトリの `.claude/skills/` フォルダ内に構築し、エージェントが自律的に発見・利用できるようにすること。

## 5. Geminiネイティブ機能のパラダイムシフト

旧来の「プロンプトを投げて返事を待つ」だけの設計は禁ずる。

1.  **Context Cachingの徹底**: 機体の基本人格、物理ルール、膨大な「過去の戦歴・修行ログ」は、毎フレーム送信するのではなく、ゲーム起動時に一度だけContext Caching機能を用いてキャッシュ領域にロードすること。これにより初動トークン生成（TTFT）を爆速化せよ。
2.  **Multimodal Bidi-streamingの完全統合**: 音声だけでなく、スマホカメラのフレーム映像を `LiveRequestQueue` に同時に流し込み、AIに現実の状況やプレイヤーの体の動きをリアルタイムに理解させること。

## 6. 今後の拡張構想 (Future Extensibility from Awesome ADK Agents)

[Awesome ADK Agents](https://github.com/Sri-Krishna-V/awesome-adk-agents) の知見に基づき、今後のプロジェクト成長において以下のアーキテクチャ・パターンの導入を検討・推奨する。

1. **A2A (Agent-to-Agent) プロトコルの導入**: 他のマルチエージェントシステム（例: 観客側のエージェントや外部システム）と標準的なプロトコルで通信・連携する基盤の構築（`A2A_ADK_MCP` の事例参照）。
2. **Datastore / Firestore Session Serviceへの移行**: 現在の `InMemorySessionService` から脱却し、Cloud Datastore（Firestore）をバックエンドとした持続的なセッション管理サービスへと移行することで、Cloud Run等のサーバーレス環境でのステートレススケールアウトを実現する。
3. **Ralph Loop (Verify, Not Guess) パターン**: エージェントが生成した出力（例：UIコンポーネントや戦術ロジック）をそのまま適用するのではなく、実行前に外部検証を挟み、成功基準を満たすまで自律的に反復修正（イテレーション）を行う堅牢なパイプラインの構築。
4. **MCP Agent Tool Adapters**: 既存のMCPツール群を単純な関数として呼び出すだけでなく、直接ADKのTool（またはSub-Agent）としてラップ・変換させるアダプターパターンの採用。
