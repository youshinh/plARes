## §1. マテリアル三すくみ仕様

### 属性定義

| material  | テーマ         | 強み                 | 弱点                                         |
| --------- | -------------- | -------------------- | -------------------------------------------- |
| **Wood**  | 打撃吸収・自然 | 物理打撃に高い耐性   | 炎属性・斬撃に弱い                           |
| **Metal** | 高防御・重火力 | 防御力・攻撃力が高い | 電撃・酸（化学）攻撃に弱い、鈍重             |
| **Resin** | 軽量・高機動   | 速度・回避率が高い   | 重い物理打撃で装甲が割れやすい（大ダメージ） |

### ダメージ補正倍率（実装定数）

既存設計書に補正倍率の数値定義がなかったため、以下を実装定数として採用する。

```typescript
// shared/constants/battleConstants.ts
export const MATERIAL_DAMAGE_MULTIPLIER: Record<
  string,
  Record<string, number>
> = {
  // 攻撃側material → 防御側material → 倍率
  Wood: { Wood: 1.0, Metal: 0.8, Resin: 1.3 },
  Metal: { Wood: 1.3, Metal: 1.0, Resin: 0.8 },
  Resin: { Wood: 0.8, Metal: 1.3, Resin: 1.0 },
};
// 例: Metal(攻撃) vs Wood(防御) → 1.3倍ダメージ
```

相性ありの場合 **×1.3**、相性なしの場合 **×1.0**、相性不利の場合 **×0.8** とする。

### 対戦相手のmaterial取得タイミング

- **取得元**: Firebase Firestore（`users/{opponentId}/robots/{robotId}`）
- **取得タイミング**: マッチング成立時（シグナリングフェーズ）。試合中にWebSocket経由では取得しない。
- **フロントエンドの扱い**: インメモリにキャッシュし、ダメージ計算時に参照する。
- **WebSocketはイベント同期専用**。静的な初期ステータスの配信には使用しない。

## §2. EXゲージ仕様（新規定義）

既存設計書ではEXゲージの蓄積ロジックが未定義だったため、以下にゲーム設計として定義する。

### 基本パラメータ

```typescript
const EX_GAUGE = {
  MAX: 100,
  INITIAL: 0,
  // 蓄積トリガーごとの加算量
  ON_HIT_DEALT: 8, // 通常攻撃ヒットごと
  ON_HIT_RECEIVED: 12, // 被弾ごと（逆境蓄積）
  PER_SECOND: 1, // 時間経過（毎秒）
};
```

### 蓄積ロジック

| トリガー       | 加算量 | 備考                               |
| -------------- | ------ | ---------------------------------- |
| 通常攻撃ヒット | +8     | `critical_hit` イベント発生時は+16 |
| 被弾           | +12    | 逆境時に蓄積が早まる設計           |
| 時間経過       | +1/秒  | バトル中常時加算                   |

### 発動条件

1. EXゲージが100（MAX）に達する
2. バックエンドが対応する必殺技フレーズを生成しフロントに送信（WebSocket `event: "special_ready"`）
3. フロントエンドが画面に必殺技名UIを表示し、プレイヤーの発声を待機
4. プレイヤーが発声 → `State.CASTING_SPECIAL` へ遷移

### 状態管理

EXゲージはインメモリ（useFSMStore）で管理し、試合終了時のコミット時にのみFirestoreへログを保存する。

## §3. 開発フェーズ・優先度ロードマップ

### Phase 1（MVP）: ローカルフレンド対戦

**最優先で実装すること。**

- URL/QRコードによる即時マッチング（Firebase Firestoreシグナリング）
- 同一空間（リビング等）でのWebXR座標系共有（Cloud Anchors利用）

- 観客スマホからのQR乱入＋現実融合クラフト介入
- Web Speech APIによるローカル音声コマンド
- 詠唱（滑舌判定）システム

**理由**: ゲームコアUX（観客介入・詠唱・ローカル音声）を最速で検証するため。

- Firestoreランクスコアによる自動マッチメイキング
- 「仮想アリーナ重畳方式」（ホスト部屋の3DメッシュをゲストにAR投影）
- レーティングシステム・リーダーボード

## §4. 通信アーキテクチャ一覧（確定版）

3経路の厳密な分離は必須。**WebSocketを音声バイナリ配信に流用することは禁止。**

| 経路                        | プロトコル                 | 用途                    | 実装                |
| --------------------------- | -------------------------- | ----------------------- | ------------------- |
| 機体座標・アクション同期    | WebRTC Data Channel（UDP） | P2P低遅延（10〜30ms）   | `rtcService.send()` |
| AI音声・映像ストリーム      | WebRTC Audio/Video Stream  | Gemini Native Audio配信 | Pipecat OSS or 同等 |
| ゲームイベント通知          | WebSocket（TCP）           | JSON軽量イベント同期    | `wsService.send()`  |
| 初期ステータス取得          | Firebase Firestore REST    | マッチング時1回のみ     | `getDoc()`          |
| シグナリング（SDP/ICE交換） | Firebase Realtime DB       | P2P接続確立             | Firebase SDK        |

### Pipecat/Daily の位置づけ

- **有償SaaSプランへの依存は不要**。OSSライブラリとして統合する。
- 代替として独自WebRTCメディアサーバーの実装も可。
- ただし**WebSocketへの音声バイナリ相乗りは禁止**（Head-of-Line Blockingによるイベント遅延が発生するため）。

## §5. 観客接続仕様

### 想定規模

- **主想定**: 家族・友人 **数名**（3〜5人）
- 大規模配信（数十〜数百人）はシステムの主想定外。レートリミットによる保護を実装する。

### 接続アーキテクチャ

- 観客は**対戦プレイヤーと完全に同じWebSocketルーム**に接続する
- アリーナURLにアクセスするだけで自動入室（アプリ不要、Firebase匿名ログイン）
- 観客からの現実融合クラフトリクエストは同一WebSocketルームにブロードキャストされる

### 観客のイベントフロー

```
観客: 写真撮影 + テキスト入力
  → WebSocket送信 (event: "item_drop_request")
  → バックエンド: Nano Banana Pro で1〜2秒テクスチャ生成
  → WebSocket ブロードキャスト (event: "item_dropped", texture_url, stats)
  → 全クライアントのThree.jsがAR空間にアイテムを落下描画
```

## §6. HPとダウン判定の実装定数

### HP計算式

```typescript
const calcMaxHP = (vit: number) => 100 + vit * 2;
// vit=40 → MAX_HP=180, vit=99 → MAX_HP=298
```

### ダメージ計算式

```typescript
const calcDamage = (
  attackerPower: number,
  attackerMaterial: string,
  defenderMaterial: string,
  isCritical = false,
) => {
  const base = 10 + attackerPower * 0.3; // power=65 → base=29.5
  const multiplier =
    MATERIAL_DAMAGE_MULTIPLIER[attackerMaterial][defenderMaterial];
  return Math.floor(base * multiplier * (isCritical ? 2.0 : 1.0));
};
```

### ダウン判定

- クリティカル受けた際、`vit`をダウン耐性として判定
- ダウン確率 = `max(0, 0.5 - vit / 200)` → vit=100で確率0%

### ヒートシステム（逆転ボルテージ）発動条件

```typescript
const isHeatActivated = (selfHP: number, maxHP: number, opponentHP: number) =>
  selfHP / maxHP <= 0.2 && opponentHP - selfHP > maxHP * 0.3;
// 自HP20%以下 かつ HP差が最大HPの30%以上
// 発動時: speed一時的+20%, adlibSkill一時的+30%
```
