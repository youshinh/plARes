「プラレスAR」のシステム開発において、データ永続化とAIの記憶管理（Memory Bank）、およびAPIコストの最適化を担う「6. データ構造・Memory Bank統合設計書」を作成します。  
システム開発者の視点から、コード実装を前提とした具体的なデータベース・スキーマと、Geminiのネイティブ機能（Context Caching、MCP）を活用したアーキテクチャを詳細に解説します。

# 6\. データ構造・Memory Bank統合設計書

**対象モジュール**: モジュールD (データ・記憶基盤)**目的**: Firestoreのサブコレクション階層（users/{userId}/robots/{robotId}等）と、APIコスト破綻を防ぐ「試合中のインメモリ状態管理（State Management）」の分離を定義します1。また、Context Cachingによる初動トークン生成（TTFT）の高速化と、Managed MCP servers for Firestoreを用いたAIの自律的データベース検索の権限付与を設計します1。

## 1\. Firestoreデータベース・スキーマ設計（正規化）

データの肥大化を防ぎ、効率的な読み書きを実現するため、ルートコレクションとして users を配置し、関連データをサブコレクションに正規化して格納する階層構造を採用します2。

* **ユーザー基本情報と長期記憶** (users/{userId})  
* profile: ユーザー名（playerName）や総試合数（totalMatches）などの基本プロファイルを保持します2, 3。  
* aiMemorySummary: LLMが数試合ごとに過去ログを要約し、上書き更新し続ける機体の長期記憶サマリー（テキストデータ）を保持します2。  
* **機体ステータス** (users/{userId}/robots/{robotId})  
* material, level, stats (power, speed, vit): 機体の材質や基本物理パラメーターを定義します3。  
* personality: talkSkill（愛嬌）、adlibSkill（機転）、tone（性格・口調）といった、AIの振る舞いや実況に影響するソフトウェア層の特性を保持します3。  
* network: syncRate（シンクロ率）など、プレイヤーとの絆を示すパラメーターを保持します3。  
* **試合・修行履歴ログ** (users/{userId}/matchLogs/{matchId} および trainingLogs)  
* ドキュメントIDをタイムスタンプとし、1試合（または1回の修行）を1ドキュメントとして無限にスケール可能な構造とします2。  
* 勝敗結果（result）や対戦相手の情報に加え、AIが抽出したハイライト（例：「プロンプト武器『ネギの剣』でトドメを刺した」）を配列として保存します2, 4。

## 2\. 試合中のインメモリ状態管理（State Management）

リアルタイムのアクションゲームにおいて、毎秒数十回の座標同期やダメージ計算をFirestore等の永続化データベースに直接書き込む設計は、APIの読み書きコストと通信レイテンシの破綻を招きます5, 6。そのため、永続化層とインメモリ層を明確に分離するアーキテクチャを実装ルールとして徹底します5。

* **試合開始時の初期化 (Initialize)**: 試合開始時に、Firestoreから aiMemorySummary（長期記憶）と直近の matchLogs を取得し、Agent Development Kit (ADK) の LiveRequestQueue を通じてGemini Live APIのシステムプロンプトの初期コンテキストとして投入します4。  
* **試合中のストリーミング (In-Memory Streaming)**: 試合中（数分間）のダメージ発生、機体の移動、観客のアイテム投下などの高頻度なイベントは、ADKのローカルメモリ（またはRedis等のセッション管理機構）とWebRTCのP2P通信内でのみ状態保持（State Management）を行い、Firestoreへの書き込みは一切行いません4, 6。  
* **試合終了時のコミットと要約 (Commit & Summarize)**: 試合終了のタイミングで、インメモリに蓄積された全イベントログからGeminiに試合のハイライトを抽出させます4。抽出したハイライトを matchLogs に新しいドキュメントとして追加（addDoc）し、さらに既存の aiMemorySummary と今回のハイライトをGeminiに渡して「新しい長期記憶サマリー」を生成させ、Firestoreのドキュメントを一度だけ更新（Update）します4。この「Memory Bankアーキテクチャ」により、機体は過去の膨大な思い出を保持しつつ、Firestoreの読み書きコストを最小限に抑えることができます4。

## 3\. Context Cachingによるコンテキスト最適化

機体の基本人格、物理ルール（材質の三すくみ等）、および数千試合分の「戦歴・修行ログ」といった長大なプロンプトを、APIコールのたびに毎フレーム送信する設計は非効率であり禁じられます7, 8。

* **キャッシュのロード**: ゲーム起動時やストリーミングセッションの開始時に、これらの膨大なコンテキスト情報を Vertex AI の Context Caching 機能を用いて一度だけキャッシュ領域にロードし、キャッシュIDを発行します7, 8。  
* **TTFTの爆速化**: プレイヤーの音声入力に対して、AIはこのキャッシュされた記憶を瞬時に参照しながら応答します8。これにより、入力トークンコストを劇的に（最大70%以上）削減しつつ、AIの初動トークン生成（TTFT: Time to First Token）を爆速化させ、遅延のない深いコミュニケーションを実現します8, 9。

## 4\. Managed MCP servers for Firestoreによる自律アクセス

開発者がアプリケーション側でデータベースから過去のログを検索し、LLMのプロンプトに注入するといった従来の手動API連携を廃止します8, 10。

* **権限の直接付与**: Google Cloudの「Managed MCP servers for Firestore」を導入し、Model Context Protocol (MCP) を通じてAIエージェント自身にFirestoreへのクエリ実行権限（ツール）を付与します8, 11。  
* **動的コンテキストの自律引き出し**: これにより、対話の中で過去に関連する話題が出た際（例：以前負けた相手との再戦時など）、Geminiが自身の判断でMCPを経由してFirestoreの matchLogs や aiMemorySummary を自律的に検索・特定します8。「マスター、相手は前の春巻きの恨みを根に持ってるぜ！」といった文脈を踏まえた高度なエージェント挙動を、コードレスに近い形でシステムに組み込むことが可能となります8。
