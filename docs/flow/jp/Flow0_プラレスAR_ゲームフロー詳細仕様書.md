# **プラレスAR：ゲームフロー詳細仕様書**

[English Version (EN)](../Flow0_Game_Flow_Detailed_Specifications.md)

本仕様書は、現実空間（WebXR）と大規模言語モデル（Gemini 3.1 Pro / Live API）を融合させたAIエージェント育成型ARゲーム「プラレスAR」のコアとなるゲームサイクルを、4つのフェーズに分けて定義する。

## **Phase 1: オンボーディング（機体の「誕生」とパイロット契約）**

単なるアバター作成ではなく、プレイヤーのパーソナリティを反映した「世界に一つだけのAI相棒」を生成し、現実空間に召喚するフェーズ。

### **1.1 ハイブリッド多言語UIの初期化**

- **フロー**: アプリ起動時、ブラウザの navigator.language を読み取り、UI言語を自動設定する。
- **システム**: 静的なメニュー項目（「設定」「試合開始」など）はローカルのJSON辞書で描画し、以降のAI実況や生成される字幕は、この言語設定に基づきバックエンドから動的（Server-Driven UI）に描画される。

### **1.2 機体のマルチモーダル生成**

- **カメラスキャン**: プレイヤーがインカメラで顔や表情をスキャンする。Gemini 3.1 ProのVisionモデルがその「オーラや骨格」を解析し、初期ステータス（パワー、スピードなど）とメインカラーを決定する。
- **フォールバック**: 顔スキャンをスキップした場合は、テキストプロンプト（例：「熱血漢でスピード重視のロボット」）による生成を受け付ける。
- **初期人格（Personality）の決定**: この時点での対話や解析結果から、初期の tone（口調・性格、例：「生真面目な武士」）と、talkSkill（愛嬌）などのソフトウェアパラメーターが決定し、Firestoreに保存される。

### **1.3 初召喚と空間認識（WebXR）**

- **フロー**: プレイヤーがスマホカメラを現実のテーブルや床（あるいは自作の木工作品）に向ける。
- **システム**: WebXRの Hit Test API と Depth Sensing API により空間が3Dメッシュ化され、平坦な場所に機体がAR表示される。ここで初めて、機体とプレイヤーの「Live接続」が開始され、音声による挨拶が交わされる。

## **Phase 2: 日常の共創（散歩・修行モード）**

バトル以外の日常的なコミュニケーションを通じて、AIエージェントの「記憶（コンテキスト）」と「愛着」を育むフェーズ。

### **2.1 散歩モード（お出かけ・探索）**

- **自律移動**: 機体は取得した空間メッシュ（NavMesh）上を自律的に歩き回り、段差があれば避けたり登ったりする（通信ラグゼロのローカル処理）。
- **Vision連動とプロアクティブ対話**: スマホのアウトカメラ映像が毎秒1〜2フレームでGemini Live APIにストリーミングされる。AIは映像を監視し、例えば「DIYで作った木製家具」を認識すると、「マスター、この木材いい匂いがするな！」などと自発的（Proactive Audio）に実況・対話を開始する。
- **現実融合クラフト**: 散歩中、現実の物体（例：食卓の「揚げ春巻き」）をカメラで撮影し、「レーザーを放つ聖剣」というテキストとともに送信。バックエンドの Nano Banana Pro が画像融合（8-Image Mix）を行い、1〜2秒で「春巻きの質感を持った3D武器テクスチャ」を生成し、インベントリに保存する。

### **2.2 修行モード（詠唱特訓）**

- **フロー**: 迫りくる障害物を避けながら、必殺技の「詠唱（早口言葉）」の精度を高めるミニゲーム。
- **滑舌判定エージェント**: プレイヤーの生の音声波形（Native Audio）をGeminiが直接解析し、テキストの「正確さ」、完了までの「スピード」、そして声の大きさや震えなどの「気迫」の3軸でスコア化する。このスコアが高いほど、機体との「シンクロ率」が上昇する。

## **Phase 3: 実戦（AR対戦マルチプレイ）**

育成した機体を信じ、セコンドとして指示を出しながら共に戦うメインコンテンツ。

### **3.1 AR空間の同期とホログラム重畳**

- **フロー**: ホストとゲスト（対戦相手）がマッチングする。
- **システム**: ホスト側（例：大阪のリビング）の地形データ（障害物の3Dメッシュ）がゲスト側に送信され、ゲストの画面にはホスト側の障害物が「半透明のホログラム」として重畳表示される。これにより、異なる物理環境下でも公平な対戦アリーナが構築される。

### **3.2 3層優先度制御による自律戦闘**

試合中、機体は以下の3層のステートマシン（FSM）により、プレイヤーの指示と自律思考をブレンドして動く。

1. **【優先度 低】反射層（ローカル演算）**: 敵の周囲を旋回する、隙を見て接近するなど、WebXRのNavMeshに沿った通信ラグゼロの自律移動。
2. **【優先度 中】戦略層（クラウドAI）**: 数秒に1回、AIが戦況を分析し、「右へ回り込め」「あの障害物の裏へ隠れろ」といったマクロな方針（Function Calling）を決定し、機体のステートを書き換える。
3. **【優先度 高】戦術層（プレイヤーの割り込み）**: プレイヤーがUIをタップ、または音声（Web Speech API）で「避けろ！」と叫んだ瞬間、クラウドの判断を全てキャンセルし、物理インパルスを加えて緊急回避を行う（レイテンシ・ハイディング）。

### **3.3 観客介入と「拒絶行動」**

- **アイテム投下**: 観客（例：娘さんたち）が別のスマホからWebブラウザ経由で乱入し、現実融合クラフトで作った変なアイテム（例：ピンクのリボン付きチェーンソー）をARアリーナに投下する。
- **機体の性格による判断**: 機体のAI人格が「硬派でプライドが高い」場合、AIはこれを拾うことを拒絶（reject_item 関数を発行）し、「こんなフザケた武器で戦えるか！」と文句を言いながら蹴り飛ばすモーションを実行する。

## **Phase 4: 成長と進化（Memory Bank統合）**

戦いの結果を単なる数値の増減ではなく、「記憶」と「外見の変化」として機体に刻み込むフィードバック・フェーズ。

### **4.1 記憶の要約と定着（Memory Bank）**

- **フロー**: 試合終了時、バックエンドは試合中（インメモリ）の全アクションログをGeminiに渡し、ハイライトを要約させる。
- **システム**: 「春巻きソードで逆転した」「何度も緊急回避に成功した」といった文脈メタデータが、Firestoreの長期記憶ドキュメント（aiMemorySummary）にマージ・更新される。

### **4.2 プロシージャル・テクスチャ進化**

- **視覚的フィードバック**: 一定試合数をこなすなど特定の節目（マイルストーン）で、Nano Banana Proのバッチ処理が裏側で走る。過去の記憶（aiMemorySummary）に基づき、「激しい炎の攻撃を受けた黒い焦げ跡」などの戦歴が、現在の機体のテクスチャに動的に合成（上書き）される。

### **4.3 アンビエントな成長フィードバック**

- **日常への還元**: 後日、再び「散歩モード」で穏やかな風景（夕日の沈む公園など）を歩いている際、AIがMCP（Model Context Protocol）を通じて自律的に過去の記憶を検索する。
- **発話例**: 「……そういえば、1ヶ月前のあの激闘の時、マスター焦ってすっ転びそうになってたよな。……俺たち、少しは強くなったかな。」といった、成長を実感させるエモーショナルな対話が突発的に発生する。
