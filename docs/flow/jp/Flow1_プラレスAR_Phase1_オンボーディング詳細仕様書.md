# **プラレスAR ゲームフロー詳細仕様書**

## **Phase 1: オンボーディング（機体の「誕生」とパイロット契約）**

[English Version (EN)](../Flow1_Phase1_Onboarding_Detailed_Specifications.md)

本仕様書は、「プラレスAR」におけるアプリ初回起動から、独自のAIエージェント（機体）を生成し、現実のAR空間へ初召喚するまでのシステムフローとデータ処理構造を定義する。

### **1\. ユーザーエクスペリエンス（UX）目標**

- **「相棒感」の創出**: 単に用意されたアバターを選ぶのではなく、プレイヤー自身の情報（顔の骨格、表情、あるいは言葉）から機体が「抽出・生成」される体験を提供する。
- **摩擦のない導入**: ネイティブアプリのインストールを不要とし、ブラウザ（WebAR）から数タップでシームレスに3D空間へ移行させる。
- **プライバシーの配慮**: カメラ撮影に抵抗があるユーザーのためのフォールバック（代替手段）をUIとして自然に組み込む。

### **2\. 画面遷移とUI制御フロー**

画像(693, 694, 695)のUIモックアップをベースとした、フロントエンド（React）における画面遷移フローを定義する。

#### **Step 1: 言語の自動解決と選択画面 (Ref: 693.png)**

1. **システム起動**: ユーザーがURLにアクセス。
2. **自動解決**: フロントエンドは navigator.language を読み取り、ユーザーのブラウザ言語（例：ja-JP）を取得。
3. **UI表示**: 「表示言語を選択 (Choose your language)」のモーダルを表示。デフォルトで自動取得した言語がハイライトされている。
4. **データ永続化**: 選択された言語（ja, en, es 等）はローカルストレージに保存され、以降の静的UI（設定メニュー等）のi18n辞書の切り替え、およびバックエンドのAgent Development Kit (ADK) への初期コンテキストとして送信される。

#### **Step 2: プラレスラー生成画面 (Ref: 694.png, 695.png)**

1. **カメラパーミッション**: ブラウザの getUserMedia APIをコールし、インカメラの権限を取得。
2. **UI表示**: 画面中央にインカメラの映像をリアルタイム表示。「📸 撮影する」ボタンを配置。
3. **モデル選択**: 画面下部のラジオボタン（Model A / Model B）で、ベースとなる機体の骨格タイプ（重量級/軽量級など）を選択可能にする。
4. **スキップ導線**: 画面左下に「スキップ→」ボタンを配置し、プライバシー配慮のフォールバックルートへの導線を確保する。

### **3\. 機体のマルチモーダル生成エンジン（バックエンド処理）**

ユーザーのアクション（撮影 or スキップ）をトリガーに、バックエンドの`gemini-3.1-pro-preview` (Visionモデル) が機体の初期ステータスと人格を決定する。

#### **3.1 カメラ撮影ルート（Main Route）**

1. ユーザーが「撮影する」をタップ後、「ロボット召喚！」ボタンを押下。
2. フロントエンドは、撮影された顔画像のBase64データと、選択されたモデルタイプ（A or B）をバックエンドAPIへPOST送信。
3. **Gemini Visionプロンプト実行**:
   - _System Prompt_: 「あなたは熟練のロボットエンジニアです。入力された人物の顔画像から『オーラ』『骨格の力強さ』『表情の豊かさ』を解析し、この人物をマスターとするプラレスラーの初期ステータスを生成してください。」
   - _Output Schema_: 指定したJSONスキーマでの出力を強制。

#### **3.2 スキップルート（Fallback Route）**

1. ユーザーが「スキップ→」をタップ。
2. カメラ映像領域がテキスト入力エリアに切り替わる。
3. **プロンプト入力**: ユーザーに「どんな機体にしたいか？（例：熱血漢でスピード重視の木製ロボット）」を自然言語で入力させる。
4. **Gemini Textプロンプト実行**: 入力テキストを元に、撮影ルートと全く同じJSONスキーマで初期ステータスを生成する。

#### **3.3 初期パラメータ（JSON）の決定とFirestore保存**

Geminiから返却されたJSONデータをパースし、Firestoreの users/{userId}/robots/{robotId} 配下へ初期データとして保存する。

**【生成される主要パラメータ群】**

- **ハードウェア（物理特性）**
  - material: ベース材質（例：wood, metal, resin）。DIY好きのマスターであれば「打撃に強い木材（Wood）」が選ばれやすいといった重み付けも可能。
  - stats: power (物理), speed (機動), vit (耐久)。合計値は一定（例：100）にキャップ（制限）される。
  - baseColor: メインカラーのHEXコード。
- **ソフトウェア（AI人格特性）**
  - tone (口調・性格): 例「関西弁の熱血漢」「クールで理知的な武士」。今後のTTS（音声合成）のベースとなる最重要プロンプト変数。
  - talkSkill (愛嬌・トーク力): 0〜100。観客のアイテム投下確率（ヘイトコントロール）に影響する隠しステータス。
  - adlibSkill (機転): 0〜100。環境認識時の自発的発話（Proactive Audio）の頻度と内容のぶっ飛び具合を制御。

### **4\. 初召喚と空間認識 (WebXR起動)**

機体のデータがFirestoreに作成された直後、AR空間への移行とファーストコンタクトが行われる。

#### **4.1 ARセッションの開始**

1. 画面が暗転し、アウトカメラの映像（WebXRセッション）に切り替わる。
2. **空間認識**: Hit Test API と Depth Sensing API がバックグラウンドで起動。
3. 画面中央にレティクル（ターゲットマーカー）を表示。ユーザーがカメラを現実の床やテーブル（フローリングや自作の木工作品の上など）に向けると、Hit Testが平面の座標を認識し、レティクルがピタッと吸着する。

#### **4.2 召喚演出とモデルロード**

1. ユーザーが画面をタップ（配置位置の確定）。
2. **動的モデルロード**: Three.jsが、決定されたベースモデル（A or B）のglTFファイルを非同期ロード。同時に、Geminiが生成した baseColor をマテリアルに適用し、stats の power などの値に応じてボーンスケール（腕の太さ等）を微調整してレンダリング（Server-Driven Rendering）。
3. AR空間の指定座標に、機体が「着地モーション」と共に表示される。

#### **4.3 Live API接続とファーストコンタクト**

1. 機体が着地した瞬間、バックエンドのADKがGemini Live API（`gemini-2.5-flash-native-audio-preview-12-2025`）との双方向ストリーミング（WebSocket）接続を確立。
2. **初期コンテキスト注入**: 接続時の初回ペイロードとして、先ほど生成した tone（性格）をシステムプロンプトに注入。
3. **ファーストコンタクト（自発的発話）**: 機体がプレイヤー（カメラ）を見上げ、最初の挨拶を行う。
   - _例（Toneが「熱血漢」の場合）_: 「マスター、待たせたな！あんたのオーラ、ビシビシ伝わってきてるぜ。今日からよろしく頼む！」
   - この音声は生の音声波形（Native Audio）を`gemini-2.5-flash-native-audio-preview-12-2025`経由でストリーミングし、ラグなく機体から発せられる。

この瞬間をもってオンボーディングが完了し、機体はプレイヤーの指示待ち（基本自律ループ：State.HOVERING）状態へ移行。Phase 2（日常の共創）またはPhase 3（実戦）への準備が整う。
