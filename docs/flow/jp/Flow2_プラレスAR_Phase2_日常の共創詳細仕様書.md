# **プラレスAR ゲームフロー詳細仕様書**

## **Phase 2: 日常の共創（散歩・修行モード）**

[English Version (EN)](../Flow2_Phase2_Daily_Co-creation_Detailed_Specifications.md)

本仕様書は、「プラレスAR」において対戦以外の日常的なコミュニケーションを通じて、AIエージェントの「記憶（コンテキスト）」と「絆（シンクロ率）」を育むための2つのモード（散歩・修行）のシステムフローと実装要件を定義する。

### **1\. ユーザーエクスペリエンス（UX）目標**

- **アンビエントな存在感**: 機体を単なるゲームの駒ではなく、「現実世界を一緒に認識している相棒」として感じさせる。
- **無限のクラフト体験**: ユーザーの趣味（DIY作品など）や日常品（夕食のおかず等）を、ゲーム内にシームレスに取り込む「現地調達」の喜びを提供する。
- **生身のスキルアップ**: プレイヤー自身の「滑舌」や「気迫」をシステムが評価することで、ボタン連打ではない本当の「特訓」を体感させる。

### **2\. 散歩モード（お出かけ・探索・クラフト）**

現実空間を共に歩き、AIと対話し、アイテムを生成するモード。Gemini Live API（`gemini-2.5-flash-native-audio-preview-12-2025`）のMultimodal Bidi-streaming（双方向ストリーミング）をフル活用する。

#### **2.1 WebXRによる自律移動（ローカル処理）**

通信ラグを排除し、自然に現実世界を歩き回らせるためのフロントエンド基盤。

1. **空間スキャン**: WebXR Device API（Hit Test API / Depth Sensing API）を常時稼働させ、スマホカメラが捉えた部屋の形状を3Dメッシュ化する。
2. **NavMesh動的生成**: 取得した頂点データから、Three.js上でキャラクターが歩行可能なポリゴン領域（NavMesh）をローカル演算で動的生成する。
3. **自律アクション**: クラウドAIからの特別の指示がない状態（State.HOVERING）では、機体はこのNavMesh上を自律的に歩き回り、プレイヤーが作った木製家具の段差などを滑らかに迂回・登頂する。

#### **2.2 Vision連動とプロアクティブ（自発的）対話**

AIが現実の風景を見て、自ら話しかけてくるシステム。

1. **ストリーミング開始**: スマホのアウトカメラ映像を毎秒1〜2fpsに間引き、バックエンドのADK（Agent Development Kit）が管理する LiveRequestQueue に継続的に流し込む。
2. **コンテキスト認識**: Gemini Live API（`gemini-2.5-flash-native-audio-preview-12-2025`）が映像を監視。ユーザーの personality.adlibSkill（機転）の数値に基づき、特定の物体を検知した際にトリガーを発火させる。
3. **発話例（DIY木工作品の認識）**:
   - _ユーザーの背景_: 木材メーカー勤務でDIYが得意。
   - _AIの反応_: 部屋にある手作りの木製階段やフローリング、DIY作品をVisionが検知。
   - _自発的発話（TTS）_: 「おっ、マスター。この木材、いい匂いと手触り（質感）がするな！あんたの作品か？俺の装甲もこういう頑丈な木で補強してほしいぜ！」と、コンテキストに合わせた実況をローテンテンシで再生する。

#### **2.3 現実融合クラフト（アイテムの現地調達）**

日常のアイテムを3Dデータ化し、インベントリに保存するシステム。

1. **マルチモーダル入力**: ユーザーが現実のアイテム（例：夕食の「揚げ春巻き」）をカメラで撮影。テキストボックスに「これをレーザーを放つ伝説の聖剣にして」とプロンプトを入力し送信。
2. **8-Image Mix（画像融合）**:
   - バックエンドの Nano Banana Pro (`gemini-3.1-flash-image-preview`) APIをコール。
   - 春巻きのパリパリとした「質感・形状」と、聖剣の「概念」を融合させ、1〜2秒で新しい3Dテクスチャ（あるいは2.5Dビルボード画像）を動的生成する。
3. **データ永続化**: 生成されたアセットURLは、Firestoreの users/{userId}/inventory に保存され、カスタマイズ画面で機体の右手ボーン等に mesh.add() で動的にアタッチ可能となる。

### **3\. 修行モード（シンクロ特訓）**

対戦に向けた「反射神経」と「滑舌（必殺技の詠唱）」を鍛えるソロ用ミニゲーム。通信コストを極小化するハイブリッド設計を採用する。

#### **3.1 ハイブリッド・ゲーティング設計（ローカル反射訓練）**

APIコストをゼロに抑えるためのフロントエンド処理。

1. **障害物回避ゲーム**: AR空間の奥から、丸太や鉄板などの障害物が機体に向かって迫りくる。
2. **ローカル音声検知**: プレイヤーが「右！」「避けろ！」と叫ぶ。ブラウザ標準の Web Speech API が遅延ゼロで音声をテキスト化。
3. **ステート強制上書き**: テキストを検知した瞬間、フロントエンドの3層優先度制御ステートマシンの「Priority 1（最高優先度）」が発火。機体に物理インパルスを加え、ラグゼロで回避アクションを実行させる。

#### **3.2 滑舌判定エージェント（Native Audio解析）**

必殺技の詠唱精度を判定し、シンクロ率を高めるコアシステム。

1. **詠唱トリガー**: 画面にランダムな早口言葉プロンプト（例：「超絶熱々揚げ春巻きストライク！」）がテロップ表示される。
2. **Native Audio送信**: プレイヤーが発声した音声（16kHz PCM波形データ）を、STT（テキスト化）を挟まずに生のまま`gemini-2.5-flash-native-audio-preview-12-2025`にストリーミング送信する。
3. **3軸評価ロジック**: バックエンドの「滑舌判定エージェント」が以下の3軸でスコア（0.0〜1.0）を算出する。
   - **正確さ (Accuracy)**: 指定テキスト通りに発音できたか。
   - **スピード (Speed)**: 詠唱完了までのタイム。
   - **気迫 (Passion/Sentiment)**: 声の大きさ、震え、熱量（Native Audioならではの解析）。
4. **結果の反映**: 総合スコアに応じて、機体とプレイヤーの「シンクロ率（Sync Rate）」が上昇。この数値はFirestoreの機体データに更新（Update）され、実戦時の「クリティカル発生率」や「AIのプレイヤー指示への従順さ」に直結する。

#### **3.3 レイテンシ隠蔽システム（UXの担保）**

滑舌判定のAPI通信（1〜2秒のラグ）をプレイヤーに気づかせないための実装。

1. プレイヤーが詠唱を終えた（またはマイクボタンを離した）瞬間、フロントエンドはAPIの判定結果を待たずに、即座に約3秒間の「機体がオーラを溜める詠唱アニメーション（State.CASTING_SPECIAL）」を開始する。
2. アニメーションが終了する前にバックエンドから判定JSONを受信。
3. アニメーション終了フック（onAnimationEnd）で結果を評価し、成功ならド派手な攻撃エフェクトを描画し、失敗（滑舌が悪かった等）なら機体がズッコケるモーションへ分岐させる。

### **4\. 記憶の定着（Phase移行への準備）**

散歩や修行モードが終了したタイミングで、これらの体験をAIの血肉とする。

1. バックエンドのPythonサーバーは、セッション内のイベントログ（例：春巻きの剣を作った、詠唱で気迫スコアが高かった）を`gemini-3.1-pro-preview`に渡し、要約（Summarize）させる。
2. 要約されたテキストは、Firestoreの長期記憶ドキュメントである aiMemorySummary にマージ更新される。
3. これにより、次回の対戦（Phase 3）や次回の散歩で、機体が「この前の修行みたいに気合い入れていこうぜ！」と文脈を踏まえた発話を行う下地が完成する。
