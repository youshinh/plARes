# plARes：AIエージェント開発マスター設計指示書

[English Version (EN)](../ai_instructions.md)

**対象**: 本プロジェクトに参画するAI開発エージェント
**目的**: 本指示書は、WebARバトルゲーム「**plARes**（プラレス）」のバックエンドAI（マルチエージェント・ストリーミング・サーバー）を実装するエンジニアへのマスタープロンプトです。詳細な役割定義や共創ルールについては、ルートディレクトリの [AGENTS.MD](../AGENTS.MD) を真（Source of Truth）として遵守してください。

## 1. プロジェクト・ビジョンとUX定義

本システムは、現実の空間（机の上やリビングの木材フローリング、自作の木工作品など）を舞台に、自分だけのオリジナルAIロボット（プラレスラー）を召喚し、声とジェスチャーで指示を出して戦わせるメタバース・ゲームである1。単なる「強い機体の殴り合い」ではなく、競技性と「モノづくり（DIY）」の喜びを融合させ、さらに観客（オーディエンス）が専用アプリ不要でブラウザ経由から乱入し、ゲーム体験を拡張する高度なエンタメ性を提供する1。

## 2\. エージェント行動規約・開発指針 (Core Principles)

AI開発エージェントは、以下の原則に従って自律的にコードを設計・実装すること。

- **Developer Knowledge MCP Serverの常時接続**: コード実装にあたり、最新のGoogle Cloud APIの仕様やベストプラクティスについて不明点が生じた場合は、手探りで実装するのではなく、自律的に「Developer Knowledge MCP server」へ接続し、公式ドキュメントをリアルタイムに参照しながら技術的課題を解決すること2, 3。
- **コスト極小化とDynamic Gating**: AIのAPIランニングコストを抑えるため、重い推論や生成処理（映像ストリーミング、Veo 3.1、Lyria API等）は常時稼働させず、「必殺技発動時」や「5試合に1回のマイルストーン」といったイベント駆動型（特定のトリガー時のみ実行）で制御するダイナミック・ゲーティング設計を徹底すること3。
- **レイテンシ・ハイディングの徹底**: AIの推論（1〜2秒）による通信ラグを、アクションゲームのUX上で完全に隠蔽する「モーション先行型」の非同期ロジックを必ず実装すること3。必殺技の発声トリガーと同時にフロントエンドで約3秒のチャージアニメーションを即座に開始させ、その描画の裏でAPI通信と判定処理を並行して完了させること3。
- **Geminiネイティブ機能の解放**: 単なるテキスト・チャットボットとしての実装は禁ずる3。旧来の「プロンプトを投げて返事を待つ」だけの設計ではなく、「視覚・聴覚の双方向ストリーミング（Native Audio/Vision）」「プロンプト入力コストを削減する記憶のキャッシュ（Context Caching）」「自律的ツール実行（MCP）」をフル活用した真のエージェント・アーキテクチャを構築すること3, 4。

## 3\. 技術スタックと使用クラウドサービス

システムは、高頻度な通信と描画処理をフロントエンド（エッジ）とP2Pで処理し、重い推論・戦略判断をGCP側へオフロードする非同期ハイブリッド設計を前提とする5。

- **フロントエンド（モジュールA: 体・感覚）**
- **UI/フレームワーク**: React5。
- **3D/ARレンダリング**: Three.js, WebXR Device API (Depth Sensing, Hit Test)5。
- **超低遅延ローカル入力**: Web Speech API (OS標準の音声認識を利用した遅延ゼロの基本操作)5。
- **バックエンドAI（モジュールB: 脳・戦略）**
- **言語/フレームワーク**: Python, Agent Development Kit (ADK) / Antigravity5。
- **コアAIモデル**: Vertex AI Gemini 3.1 Pro5。
- **双方向ストリーミング**: Multimodal Live API (Bidi-streaming)5。
- **リアルタイム通信網（モジュールC: 神経）**
- **座標・アクション同期**: WebRTC Data Channel (UDPベースのP2P通信。ラグを10〜30msに抑制)5。
- **AI向けマルチメディア転送**: WebRTC Audio/Video Stream (Pipecat / Daily等のOSS活用)5。
- **イベント同期**: WebSocket5。
- **データ・記憶基盤（モジュールD）**

1.  **AI Native Performance**: Gemini 1.5/2.5/3.x のマルチモーダル・ネイティブ機能（Native Audio解析、Live APIインターラプション処理）を前提とし、フロントエンドの描画と並行して推論を行う「レイテンシ隠蔽設計」を徹底する。
2.  **Context Efficiency**: `performance_cost_report.md` を正とし、Context Cachingをフル活用して入力トークンコストを70%以上削減、応答速度（TTFT）を最大化する。
3.  **Autonomous Agent**: AIは単なる応答ボットではなく、Firestore MCPサーバーを通じた「記憶（Memory Bank）への自律アクセス」と「ステータス進化（Evolution）」を司る主体として振る舞う。
4.  **Cloud Native Scalability**: Cloud Run上での水平スケールを考慮し、セッション管理はステートレスに行い、メディアストリーミングは Live API 直結または軽量な仲介に絞る。

- **Nano Banana Pro (Gemini 2.5 Flash Image)**: 8-Image Mixによる現実融合アイテムクラフト5。
- **Veo 3.1 API**: 5試合ごとのマイルストーン動画生成5。
- **Lyria API**: 勝利のプロシージャル凱歌・BGM生成5。

## 4\. 開発フェーズとモジュール別実装規約

システム全体を一度に構築するのではなく、以下のフェーズに分割して段階的に実装を進めること2。

### フェーズ1：フロントエンドと通信網の基盤構築

- **AR空間認識の実装**: スマホカメラから現実空間の床や自作の木工作品などの障害物をスキャンし、Depth Sensing APIやHit Test APIを用いてリアルタイムに3Dメッシュ化と空間アンカーを生成する処理（ローカルの反射神経）を記述すること6, 7。
- **超低遅延通信（P2P）の実装**: プレイヤー間の座標同期や基本アクションのトリガー用に、ラグを10〜30msに抑え込むWebRTC Data Channelの直接通信を実装すること6, 7。

### フェーズ2：バックエンドAIの双方向ストリーミング実装

- **キューイングと並行処理**: Python環境下で、セッションごとにスレッドセーフな非同期バッファである LiveRequestQueue を生成すること6, 7。
- **ノンブロッキング制御**: asyncio.gather() を用いて、上りストリーム（スマホからの映像フレームや音声の流し込み）と、下りストリーム（runner.run_live()による音声出力やJSON形式のFunction Calling発行）を同時に走らせる並行タスク制御を厳密に実装すること6, 7。AIの処理中であってもプレイヤーの割り込み（Interruption）をブロックしない設計とすること6, 7。

### フェーズ3：データ・記憶基盤と「MCP直結」の構築

- **インメモリ状態管理**: 高頻度なRead/WriteによるFirestoreのコスト破綻を防ぐため、試合中のリアルタイムな状態管理（State Management）はADKのローカルメモリやセッション内でのみ保持すること8, 9。
- **試合終了時の要約と保存**: 試合終了のコミット時にのみ、Geminiに試合イベントからハイライトを抽出させ、長期記憶のサマリーテキスト（aiMemorySummary）としてFirestoreを一度だけ更新すること10。

## 5\. 重要なパラダイムシフト（Geminiの真価を発揮するための要件）

エージェントは、本システムを商用レベルへ引き上げるために、以下のGeminiネイティブなアーキテクチャ要件を必ず満たしたコードを実装すること10。

1. **映像(Video)を含めたMultimodal Bidi-streamingの完全統合**音声だけでなく、スマホカメラのフレーム映像（毎秒1〜2fps）を LiveRequestQueue に同時に流し込むこと11。これにより、AIに「現実の部屋の状況」や「プレイヤーの体の動き」をリアルタイムに理解させる11。プレイヤーの回避動作にシンクロしてシールドを展開したり、観客のピースサインに反応したりといった次元の違うインタラクションを実装すること11。
2. **Context Cachingによるコンテキストの最適化**毎回のプロンプトに前提条件を送信する設計は非効率であるため禁止する11。機体の基本人格、物理ルール、および膨大な「過去の戦歴・修行ログ」は、ゲーム起動時に一度だけ Vertex AI の Context Caching 機能を用いてキャッシュ領域にロードすること11, 12。これにより入力トークンコストを劇的に削減し、初動トークン生成（TTFT）を爆速化させること11, 12。
3. **Managed MCP servers for Firestoreによる自律アクセス**アプリ側でFirestoreからデータを読み込んでAIに渡すのではなく、AI自身にデータベースへのクエリ実行権限（ツール）を付与すること12, 13。対話の文脈からAIが自律的に過去の試合履歴（matchLogs）や長期記憶を検索・抽出する仕組みを構築すること9, 13。
4. **Nano Banana Proによる現実融合クラフト（8-Image Mix）**単なるテキストからの画像生成ではなく、現実の写真を取り込んだクラフトシステムを実装すること14。観客のスマホから送信された現実の物理アイテムの写真（例：春巻き）とテキストプロンプトをマルチモーダル入力として受け取り、Nano Banana Proの「8-Image Mix（画像融合）」機能を実行すること14, 15。1〜2秒で即座に生成されたテクスチャを、アリーナ上の3Dオブジェクトに動的にマッピングする処理を記述すること15。
