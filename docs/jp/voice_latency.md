# 5\. 滑舌判定・レイテンシ隠蔽システム設計書

[English Version (EN)](../voice_latency.md)

**対象モジュール**: モジュールB & UI (戦闘ロジック)**目的**: 必殺技発動時におけるAIの推論やAPI通信に伴うラグ（1〜2秒）をアクションゲームのUX上で完全に隠蔽するため、フロントエンドの「モーション先行型」非同期ロジックを定義します1, 2。また、テキスト化（STT）をバイパスし、生の音声波形（Native Audio）を用いた gemini-2.5-flash-native-audio-preview-12-2025 による3軸評価（正確さ、スピード、気迫）の高度な滑舌・採点ロジックを設計します3, 4。

## 1\. 非同期「モーション先行型」レイテンシ・ハイディング設計

格闘ゲームやアクションゲームにおいて、入力から発動までに1〜2秒の遅延が発生することはUXの致命的な破綻を招きます。これを防ぐため、UIと通信を切り離すレイテンシ・ハイディング（遅延隠蔽）を実装します1, 2。

### 1.1 トリガー検知とアニメーションの即時開始

- **処理ロジック**: EXゲージがMAXになり、プレイヤーが必殺技ボタンをタップ（または発声トリガーを引いた）瞬間、フロントエンド（React / Three.js）はAPI通信の完了を待たずに、即座に約3秒間の「チャージ・詠唱アニメーション」を開始します1, 2, 5。
- **状態同期**: この詠唱開始のイベントはWebRTC Data Channel経由で即座に対戦相手にも同期され、双方の画面で機体が無防備なチャージ状態に入ります5。

### 1.2 並行ストリーミングとJSONコールバック

- **処理ロジック**: フロントエンドでのアニメーション描画と完全に並行して、バックエンドへ詠唱音声のストリーミングを開始します1, 2。
- **ラグの隠蔽**: アニメーションが再生されている3秒間の裏側で、バックエンドのGemini Live APIが判定処理を実行します6。アニメーションが終了する前に判定結果のJSON（Critical または Miss）を受信してバッファに待機させることで、ユーザーの体感上は「遅延が一切存在しない」キビキビとした操作感が実現します1, 2, 6, 7。

## 2\. Native Audioストリーミング基盤の実装

詠唱される早口言葉（例：「超絶熱々揚げ春巻きストライク」など）の判定において、OS標準の音声テキスト化（STT）による文字列比較は行いません1, 2。

- **音声波形の直接入力**: フロントエンドのマイクで取得した生の音声データ（16kHz PCM形式など）を、WebRTC Audio Stream（PipecatやDaily等のOSSを利用）経由でバックエンドに送信します3, 4。
- **キューイング**: バックエンドのAgent Development Kit (ADK) は、受信した音声データをBase64エンコードし、セッション固有の非同期バッファである LiveRequestQueue に流し込みます3, 4。
- **STTのバイパス**: これにより、音声を一旦テキストに変換するプロセスを省き、生の音声波形（Native Audio）のまま gemini-2.5-flash-native-audio-preview-12-2025 のMultimodal Live APIへ直接入力します3, 4。

## 3\. 滑舌判定エージェントの3軸評価ロジック

バックエンドに常駐する「滑舌判定エージェント」に対し、入力された音声波形から以下の3軸でスコア（0.0〜1.0）を算出し、JSONとして出力させるシステムプロンプトを定義します3, 4。

1. **正確さ（Accuracy）**: 画面に表示された難解なプロンプトテキスト（早口言葉）に対して、実際の発音がどれだけ正確であったかを判定します3, 4。
2. **スピード（Speed）**: 詠唱の開始から完了までの速度とテンポを評価します3, 4。
3. **気迫（Passion/Sentiment）**: Native Audioを利用する最大の理由です。単なる文字の読み上げではなく、音声波形に含まれる声の「大きさ」「震え」「焦り」といった熱量をAIに直接解釈させます3, 4, 8。本気で叫んだ波形の熱量を高く評価し、後述のクリティカル率へのボーナスとして加算します3, 4。

ADKの runner.run_live() による非同期ジェネレータは、これらの解析・採点結果を Function Calling のJSONフォーマットで即座にダウンストリーム出力します9, 10。

## 4\. 判定結果の適用とフロントエンド分岐処理

バックエンドから返却された判定結果（JSON）は、WebSocket等を経由して全クライアントにミリ秒単位で同期されます9, 10。

### 4.1 アニメーション終了フックと分岐

フロントエンド側では、先行再生していた3秒間のチャージアニメーションの終了イベント（onAnimationEnd 等）をフックし、受信済みのJSONデータに基づいて後続の処理を分岐させます9, 10。

- **成功（Critical）時の処理**: Three.jsのレンダラーに対し、対象の機体にド派手な炎のパーティクルエフェクトなどの描画コマンドを発行し、相手に特大ダメージを適用します9-11。同時に、Firestoreのバトルログに「必殺技成功」のイベントを記録します11。
- **失敗（Miss / 噛んだ）時の基本処理**: 自爆ダメージを適用するか、機体がズッコケるなどのペナルティモーションを再生します9, 10。

### 4.2 大喜利ペナルティ（The "Slip & Slide" System）への昇華

単なる「失敗」をエンターテインメントに昇華させるため、AIの誤認識や言い間違いをフックとしたイベント駆動処理を実装します12。

- **処理ロジック**: 詠唱音声が本来の技名と異なるテキストとして認識された場合（例：「揚げ春巻き」を「ハゲ春巻き」と言い間違えた場合）、バックエンドのADK大喜利エージェントがその単語を抽出し、画像生成API（Nano Banana Pro）へ送信します12。
- **動的オブジェクト落下**: 1〜2秒で即席の「ハゲた春巻き」の3Dテクスチャが生成され、アリーナの自機に向かって落下して物理ダメージを与えるという、爆笑のハプニング（物理的なスベり）を発生させます12。

### 4.3 シンクロ率（Sync Rate）による判定補正

システム側（インメモリまたはFirestore）で保持している、機体とプレイヤー間の絆を示すパラメーター「シンクロ率（Sync Rate）」を、判定ロジックの補正値としてバインドします9, 10。

- **処理ロジック**: シンクロ率が高いほど、音声コマンドの認識ラグや滑舌判定の許容閾値を緩和し、必殺技のクリティカル発生率をベースアップさせる計算式を組み込みます9, 10。これにより、共に特訓を重ねた機体ほどプレイヤーのプレッシャー（焦りによる噛み）をカバーしてくれるという、ゲームとしての成長曲線を表現します9, 10。

## 5\. タイムアウト・再試行・フォールバック規約（確定）

必殺技中のUX破綻を防ぐため、判定APIの応答遅延時は以下を必須実装とします。

- **演出時間**: 3.0秒（許容 2.5〜3.5秒）。
- **判定受信期限**: チャージ演出の終了時点まで。
- **再試行**: 期限前に未受信の場合のみ1回再試行（軽量リクエスト）。
- **期限超過時フォールバック**: `Miss` を確定し、試合進行を継続（待機/停止は禁止）。
- **同期処理**: フォールバック結果はWebSocketで全クライアントに配信し、片側のみ演出が進む不整合を禁止。
- **ログ記録**: `special_judge_timeout` をイベントとして記録し、運用時に閾値調整できるようにします。
